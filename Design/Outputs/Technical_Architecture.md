# 《万妖录：躺平修仙》- 技术架构与模块设计文档

本文档基于 Cocos Creator 框架（建议 3.x 版本），详细定义了《万妖录：躺平修仙》的技术架构、目录规范、核心组件设计及性能优化策略。

## 1. 整体架构概述 (Architecture Overview)

本游戏属于具有高频数据交互和海量实体渲染的割草+养成类游戏。
整体架构采用 **模块化与组件化混合设计（ECS变种思路）**，实现核心逻辑与渲染表现的高度解耦：
*   **状态与数据分离**：局内数据统一由 Data 模块维护，表现层只负责根据数据更新节点。
*   **统一对象管理**：同屏海量怪和弹幕，必须依赖全局对象池技术（Object Pooling）。
*   **事件驱动模型**：模块之间通过 EventBus（事件总线）通信，避免强耦合，例如“主角升级”仅需抛出事件，UI 和技能系统自行订阅并做出响应。

---

## 2. 标准目录结构 (Directory Structure)

游戏根目录 `assets/` 下分为以下标准层级，所有开发人员需严格遵守此规范存放资源：

*   `assets/Scripts/`: 核心业务逻辑脚本
    *   `Core/`: 基础框架和全局管理器（GameManager, EventManager, PoolManager）。
    *   `Data/`: 数值表解析、常量定义、本地存档管理器。
    *   `Gameplay/`: 局内核心战斗系统（PlayerController, Enemy, SkillBase, 抓捕系统）。
    *   `UI/`: 界面相关的视图控制脚本，按照功能分包（MainMenu, BattleUI, Outgame）。
    *   `Utils/`: 通用工具类（数学计算、随机数种子控制、日志管理）。
*   `assets/Scenes/`: 场景文件（Boot 初始化、MainMenu 局外养成、Battle 局内战斗）。
*   `assets/Resources/`: 需动态加载的独立资源（配置文件 JSON, 特殊特效）。
*   `assets/Textures/`: 静态图集和散图资源（UI图片、角色切图）。
*   `assets/Prefabs/`: 预制体资源（UI面板、各种怪物实体、法术投射物、飘字）。
*   `assets/Audio/`: 游戏音效与背景BGM。
*   `assets/Animations/`: 2D骨骼动画（Spine/Live2D文件）。

## 3. 核心模块与组件设计 (Core Modules & Components)

### 3.1 全局管理器系统 (Managers)
*   **GameManager (游戏大总管)**：维护全局状态机，负责串联游戏流程（加载 -> 进主界面 -> 战斗开始 -> 游戏暂停 -> 战斗结算）。
*   **EventManager (事件总线)**：基于发布/订阅模式的消息中心。核心事件如：`OnEnemyDeath`（怪物死亡）, `OnPlayerLevelUp`（玩家升级）, `OnBossSpawn`（Boss刷新）。
*   **PoolManager (对象池)**：掌管性能命脉。提前预热高频对象（如初级草精、法修剑气、经验球、金币）。在切出关卡时负责统一回收。
*   **DataManager (数据层)**：解析 `Config/` 下的 JSON 数值表（怪物属性表、宠物图鉴表、技能表），并使用 `sys.localStorage` 进行加密持久化存储。

### 3.2 局内战斗系统 (Gameplay)
局内采用抽象基类+子类的形式实现各个对象。
*   **PlayerController (主角控制)**：负责输入监听（单摇杆虚拟按键）、移动计算、状态管理（当前血量、无敌帧、属性Buff）、并负责吸收“无主灵气”。
*   **EnemyManager (刷怪系统)**：依据时间轴数据表动态生成怪物。支持平缓期（慢移速近战）、压力期（远程弹幕/高移速）和Boss降临等模式。
*   **SkillSystem (流派构筑系统)**：
    *   `SkillManager`: 挂载于主角身上，统筹当前局内生效的所有技能。
    *   三大流派实现：**体修流**通过添加主角碰撞体或光环实现（如金钟罩反伤）；**法修流**大量使用对象池发射子弹（如万剑归宗）；**御兽流**通过实例化跟随宠物 AI 节点实现。
    *   流派共鸣机制采用 `Tag` 统计，满足同 Tag 数量即触发终极状态。
*   **CaptureSystem (抓捕与变异系统)**：极具特色的核心模块。
    *   封装计算概率公式模块：获取目标怪物最大血量、当前血量和品质状态，结合主角的“幸运值 (LUK)”。
    *   在抛掷葫芦（动画飞行）接触目标时，根据随机数种子决定抓捕成败及“晃动三次”的视觉表现。

### 3.3 UI与表现层 (UI & Presentation)
*   **UIManager (界面堆栈管理器)**：全局 UI 层级控制，采用界面栈结构。确保例如“3选1升级面板”覆盖在游戏层上方并正确触发暂停，以及处理层层弹窗（如抓捕成功提示 -> 宠物变异提示）。
*   **VFXManager (视觉特效控制)**：屏幕震动（Screen Shake）、怪物死亡化灰动画、伤害飘字（基于 Object Pool 实现）以及各种高级变异宠物的流光着色器（Shader）控制。

## 4. 数据流转与存储机制 (Data Flow)

为了保证“躺平修仙”进度不丢失及良好的拓展性：
1.  **静态数据表**：策划配置导出的 `json` 表（例如：宠物基础成长表、技能升星表），在游戏 Boot 阶段统一反序列化进入内存缓存。
2.  **动态存档数据 (SaveData)**：包含玩家的基础境界（练气、筑基...）、天赋树加点、当前拥有的宠物、灵石资源量、洞府家具布置。
3.  **存档更新时机**：局内只在关卡结束（通关或死亡结算）时回写数据，局外在每次购买、孵化蛋、升级等行为后即时进行本地持久化。后期考虑增加云存档（防作弊与防丢失）。

## 5. 性能优化规范 (Performance & Optimization)

考虑到游戏需要同屏显示成百上千的小怪及炫酷法术，且需要适配中低端手机，以下规范必须遵守：
*   **坚决使用对象池**：所有的飞行物（法术子弹）、怪物实体、掉落物、伤害飘字，绝对不允许在局内高频使用 `instantiate` 和 `destroy`，必须从 `PoolManager` 取出和放回。
*   **物理引擎降级**：避免使用耗性能的 Box2D 刚体物理。对于割草类的大量碰撞，手写基于距离网格或四叉树（QuadTree）的简单矩形/圆形相交判断。
*   **DrawCall (合批优化)**：
    *   局内 UI 和游戏实体必须分属不同的摄像机和层级。
    *   小怪、经验球等素材必须打包在同一个图集（Sprite Atlas）中。
    *   若支持，开启 GPU Instancing 进行海量同类实体的渲染。
*   **动画降级**：怪物死亡不需要多帧倒地动画，直接缩放（Scale To 0）+ 粒子爆点替代，大幅降低内存与 CPU 动画计算开销。
*   **资源按需加载**：每个大章节的背景图、Boss 资源划分为独立的 `AssetBundle`，不使用就不加载。
